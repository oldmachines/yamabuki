Yamabuki — a SNES emulator in Zig, and the eight decisions that shape it

Yamabuki is a SNES emulator written in Zig, built to run full speed on underpowered ARM handhelds. Sixteen thousand lines, a hundred and nine commits.

⟨CLICK⟩

The emulation itself is the easy part. What's interesting is the machinery built around it to keep the thing honest on a device with no cycles to spare — and that's what this deck is about.

Performance is gated deterministically

Not by wall clock — by counters that cannot drift

.{ .path = "Rings.sfc",
   .steps      =  1582245,   // instructions retired
   .cycles     = 42884192,   // master clock

   .vram_reads = 14672028,   // renderer word fetches

   // delete the tile-row decode cache -> ~8x -> CI goes red
}

Timing a frame in CI is famously flaky. So the perf baseline doesn't time anything. It pins three counters: instructions retired, master cycles elapsed, and renderer VRAM fetches. All three are identical across Debug, ReleaseFast, and every target architecture — so the gate fails on drift rather than on noise.

⟨CLICK⟩

The clever part is the third one. VRAM reads is chosen as a proxy for the optimization it protects.

⟨CLICK⟩

Delete the tile-row decode cache and that number multiplies by roughly eight. CI goes red instantly. An optimization CI cannot see is an optimization that gets reverted by accident.

Accuracy is a comptime parameter

Spent at frame granularity, never per pixel

  • Console(cfg) is instantiated twice — fast and accurate
  • Only seven sites in the core branch on the accuracy flag
  • Every one sits at scanline or frame level — the hot loops are monomorphized
  • Runtime selection is a tagged union: one switch per frame, not per pixel

The accurate core and the fast core are the same source, instantiated twice at comptime.

⟨CLICK⟩

And only seven sites in the entire core actually branch on which one you're running.

⟨CLICK⟩

Every one of them sits at scanline or frame level, so the inner loops come out fully monomorphized — no branch, no function pointer. That's what a Cortex-A53 needs.

⟨CLICK⟩

Choosing a core at runtime is a tagged union, dispatched once per frame. The flexibility costs a switch per frame instead of a switch per pixel.

A pointer is a compile error

Save states are comptime reflection over plain data

  • The serializer walks the state tree and refuses to write a pointer
  • So derived state — page tables, decode caches, chip wiring — must be rebuilt in postLoad(), not persisted
  • It cannot silently rot: the type system won't let it
  • byteSize(T) is comptime-known → libretro's retro_serialize_size is honest

The save-state serializer walks the struct tree at comptime, and it refuses to serialize a pointer.

⟨CLICK⟩

That sounds like a limitation. It's the design. It forces every piece of derived state — bus page tables, decode caches, chip-to-ROM wiring — to be rebuilt by a postLoad hook rather than written to the file.

⟨CLICK⟩

Which means it cannot quietly go stale, because the type system won't compile the version where it does.

⟨CLICK⟩

It also makes the state size known at compile time, which is what lets libretro's serialize-size be a fixed, honest number for a whole session.

The one rule the compiler can't enforce

A Console is self-referential. Heap-allocate it. Never move it.

// yes
const c = try alloc.create(Console);
c.init(cart);          // bus page table points into c.cart, c.bus.wram
                       // the CPU holds &c.bus

// no — the page table now points into the old copy
var moved = c.*;       // undefined behaviour, silently

The whole machine is one allocation. The bus page table holds pointers into the cartridge and the work RAM inside that same struct, and the CPU holds a pointer to the bus.

⟨CLICK⟩

That's how the core gets to zero per-frame heap traffic. The price is a rule the compiler cannot check: construct it in place, and never copy the value afterwards. A moved Console has a page table pointing at its own corpse.

⟨CLICK⟩

It's documented in three separate places — which is what you do when the type system can't help you. Every other decision in this deck turns a promise into something a machine checks. This is the one that stayed a promise.

Correctness is externally anchored

The test data is fetched and revision-pinned, never vendored

  • CI resolves SingleStepTests and PeterLemon with git ls-remote at run time — no ROMs, no vectors in the tree
  • What's committed is only the hashes
  • The bar is higher than "passes": the 65816 matches per-cycle bus position over 5.12M cases
  • Every golden frame is replayed through three drivers: fast core, accurate core, libretro core

None of the test data lives in this repository. CI resolves the upstream test-vector and test-ROM repos at run time and pins them by revision.

Zero build-time dependencies is a hard constraint

Not a preference — a thing the packaging script asserts

  • SDL3 is not linked: the frontend hand-ports the ABI subset it needs and dlopens the library
  • zig build alone cross-compiles to aarch64-linux-musl on a machine with no SDL installed
  • package_handheld.sh then asserts the result is statically linked — no interpreter, no NEEDED
  • A handheld firmware won't supply the shared objects a stray dep would demand

SDL is never linked. The frontend hand-ports the slice of the C ABI it actually calls, and dlopens the library at runtime.

⟨CLICK⟩

So zig build, on its own, cross-compiles a working binary to an ARM handheld target on a machine with no SDL installed at all.

⟨CLICK⟩

And then the packaging script asserts it: no program interpreter, no NEEDED entries. Statically linked or the build fails.

⟨CLICK⟩

Because a handheld firmware will not supply the shared objects that a stray dependency would ask for. Better to fail loudly on the build machine than quietly on the device.

Shaders: compile on the laptop, not the handheld

Twelve libretro CRT presets — and no shader compiler in the binary

The CRT shaders are libretro slang presets — Vulkan GLSL. Running them the way RetroArch does means linking glslang and SPIRV-Cross into the binary — about a hundred and fifty thousand lines of C++ — and compiling shaders on the device at load.

⟨CLICK⟩

Yamabuki does the compile on the build host instead and ships the result. glslang produces SPIR-V, SPIRV-Cross produces plain GLSL in three profiles, reflection produces the uniform offsets, and the phosphor-mask PNGs are decoded to raw RGBA. The emulator therefore contains no shader compiler, no SPIR-V, and no image decoder. It reads bytes and writes them at offsets. That's the whole runtime.

slang = Vulkan GLSL. RetroArch links 150k lines of C++.

We bake it. Binary holds no compiler, no SPIR-V, no PNG.

What baking buys

  • The design goals survive — pure-Zig core, dependency-free zig build, static-musl package still passes its own assertion
  • Nothing compiles on the device: an A53 doesn't spend startup on GLSL, and a vendor compiler bug surfaces on your laptop
  • Every ambiguity — offsets, samplers, aliases, feedback, LUT dims — is resolved once, where it can fail loudly
  • A shader that can't work is absent, not broken. 31 of 36 pairs baked; the 5 skips are printed
  • The tools are built by zig c++ — even the bake step needs no toolchain the repo doesn't already pin

What that buys. First: the design goals survive. Pure Zig core, dependency-free build, and the static musl package still passes its own no-dynamic-deps assertion. A runtime shader compiler would have cost all three.

⟨CLICK⟩

Second: nothing is compiled on the device. A Cortex-A53 doesn't spend its startup budget parsing GLSL, and a bug in some vendor's shader compiler surfaces on your laptop instead of in someone's hands.

⟨CLICK⟩

Third — and this is the same idea as the perf gate — every ambiguity is resolved once, at bake time, where it can fail loudly. Uniform offsets, sampler bindings, pass aliases, feedback targets, LUT dimensions. That's why the runtime can be allocation-free after init: its job is reduced to memcpy-into-offset.

⟨CLICK⟩

Fourth: a preset appears in a profile's directory only if it actually transpiled AND every uniform in it mapped to something the runtime supplies. Thirty-one of thirty-six pairs baked. The five that didn't are printed, not swallowed — so a shader that can't work is absent rather than broken.

⟨CLICK⟩

And the tools themselves are compiled by zig c-plus-plus. Zig ships clang, so even the bake step needs no toolchain this repo doesn't already pin.
